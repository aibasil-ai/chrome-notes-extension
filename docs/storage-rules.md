# 筆記儲存規則

## 📦 儲存架構：雙軌混合儲存

本專案採用 **`chrome.storage.local`（本機）+ `chrome.storage.sync`（同步）** 的混合儲存策略。

- **Local**：主要資料來源，所有筆記都會存在這裡
- **Sync**：跨裝置同步用，受 Chrome 配額限制

---

## 🗂️ 筆記資料結構

定義於 `src/types/note.ts`：

| 欄位 | 型別 | 說明 |
|------|------|------|
| `id` | `string` | UUID，唯一識別碼 |
| `title` | `string` | 標題 |
| `content` | `string` | 內容 |
| `tags` | `string[]` | 標籤列表 |
| `createdAt` | `number` | 建立時間（ms timestamp） |
| `updatedAt` | `number` | 最後更新時間（ms timestamp） |
| `editMode` | `'plain' \| 'markdown'` | 編輯模式 |
| `pageContext?` | `object` | 選填的網頁關聯資訊（URL、頁面標題、擷取時間） |

## 💾 儲存規則詳解

### 2. 寫入（`saveNote`）— 核心邏輯

寫入時，程式會先判斷**「同步壓力」**是否存在。

**觸發同步壓力的條件（任一成立即算有壓力）：**

- **存在未同步的筆記**（包含：Local 有但 Sync 無、Local 版較新，或因單筆容量超過 7,800 bytes 而被阻擋同步的筆記）
- sync 已達到最大可同步筆數（**50 筆**）
- sync 已使用空間 ≥ **96 KB**（總配額為 102,400 bytes，預留 ~6KB）

> 💡 **單筆超大容量處理（7,800 bytes 限制）**：Chrome Sync Storage 單一 item 限制為 8,192 bytes。為了避免超出限制，系統在同步時會**自動忽略**超過 7,800 bytes 的單筆筆記（不會執行同步，將持續保持在「未同步」狀態）。但**這並不會阻擋該筆記寫入 Local**。

---

根據設定 `allowLocalSaveWhenSyncFull`（設定介面稱作「同步容量滿載時允許僅儲存本機」，但實質上是**「是否允許未同步狀態」**的寬鬆/嚴格模式開關）有兩種行為：

#### 🔴 設定關閉（預設）— 嚴格模式

**目標：維持絕對的同步一致性。當有同步壓力時，阻止可能破壞現有已同步結構的「新動作」或「擴增動作」。**

| 情況 | 行為 |
|------|------|
| 新增筆記且有同步壓力 | ❌ **直接阻擋，拋錯**（提示清理空間或開啟寬鬆模式） |
| 更新已同步的筆記，且檔案變大，且有同步壓力 | ❌ **直接阻擋，拋錯** |
| 儲存後發現有原先已同步的筆記「因總容量超出 96KB 而被暫時擠出 sync」 | ❌ **回滾到儲存前的狀態並拋錯**（保護早已同步的舊資料不被新變更擠掉） |
| 更新早已處於「未同步狀態」的舊筆記或超大筆記 | ✅ **允許更新 Local**（因為它不影響已同步資料的完整性，系統允許使用者編輯它們以期望縮小容量或修復） |

#### 🟡 設定開啟 — 寬鬆模式

**目標：以成功儲存資料為最優先。即使遇到同步壓力（空間不足或未同步累積），也要保證變更至少能先存入本機（Local）。**

| 情況 | 行為 |
|------|------|
| 新增筆記且有同步壓力 | ✅ **僅存 Local，標記為「未同步」** |
| 更新已同步筆記且變大且有同步壓力 | ✅ **僅存 Local，標記為「未同步」** |
| 更新已是未同步的筆記 | 🔄 **優先嘗試存入 sync，若不符合同步條件則 fallback 僅存 Local**（確保雲端釋出空間時能自動治癒） |

### 3. Sync Storage 的分 Chunk 規則

為克服 Chrome sync storage **單一 key 最大 8,192 bytes** 的限制，筆記同步時採用 **chunk 陣列方式打包儲存**：

```
每個 chunk key: notes_chunk_0, notes_chunk_1, notes_chunk_2 ...
chunk 大小上限: 7,800 bytes（保留 ~392 bytes 給 key 名稱和 JSON 陣列結構）
notes_chunk_count: 記錄目前 chunk 總數
```

**同步優先順序與過濾規則：**

1. **單筆容量過濾**：若單筆筆記超過 7,800 bytes，直接跳過不放入 sync 列隊（永久留在 Local 作為未同步狀態）。
2. **已同步的舊筆記享有絕對優先權**：系統優先保留原先已經同步在 Sync 內的筆記。如果舊筆記更新變大，導致總和超出 96KB，該舊筆記會**被暫時移出 Sync**（在嚴格模式下會觸發回滾防止此情況）。
3. **剩餘空間才放入新筆記**：若還有餘裕，才會將新建立或剛解除未同步狀態的筆記加入 Sync。
4. **分發 Chunk**：將排序好的名單，依次塞進 7,800 bytes 的 chunk 取代覆蓋。

### 4. 刪除（`deleteNote`）

- 從 local 刪除後，同步更新 sync storage（sync 的 chunk 也會重新計算）。

### 5. 未同步狀態追蹤（`unsyncedNoteIds`）

- 每次儲存後，比對 local 與 sync 的 `updatedAt` 時間戳來判斷哪些筆記「尚未同步或版本較舊」
- 結果儲存在 `chrome.storage.local` 的 `unsyncedNoteIds` 欄位

### 6. 設定檔（`settings`）

- **只存在 `chrome.storage.sync`**，不放 local
- 預設值：自動儲存間隔 3000ms、純文字模式、不擷取頁面資訊、顯示容量圖示

### 7. 匯入（`importNotes`）

- 只匯入 ID 不重複的筆記，已存在的筆記會被跳過
- 匯入後重新載入所有筆記並更新 sync 使用量
- ⚠️ **匯入容量防護**：由於匯入時是逐筆呼叫寫入邏輯，若系統處於「**嚴格模式**」，當匯入的資料包含超大筆記（大於 7,800 bytes），或使總容量達到 96KB 上限時，將立刻觸發「同步壓力防護」，導致**拋錯並中斷後續所有筆記的匯入**。若需強制匯入大量資料，**強烈建議匯入前先切換為「寬鬆模式」**。

## 📊 儲存限制總結

| 項目 | 限制 |
|------|------|
| Sync 總容量上限 | **96 KB**（Chrome 給 102.4 KB，預留 ~6 KB 給開銷使用） |
| 最大可同步筆數 | **50 筆** |
| 單筆最大同步容量 | **7,800 bytes**（單筆超過此大小時，將**不再進行同步**，永久轉為未同步狀態） |
| Local 儲存限制 | 瀏覽器本身無嚴格單一檔案限制。系統**不會阻擋**大於 7,800 bytes 的筆記寫入 Local，亦無嚴格筆數限制，但它們將無法上傳至 Sync。 |
